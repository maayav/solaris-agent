# Mission Planner Agent Specification

**Agent:** Mission Planner
**Type:** `mission_planner`
**Tier:** Tier 4 (Cloud)
**Model:** `qwen-3-235b-a22b-instruct-2507` (Cerebras primary) → `moonshotai/kimi-k2-instruct` (Groq fallback)
**Temperature:** 0.85
**Poll Interval:** 1000ms

---

## 1. Identity & Role

Mission Planner is the **mission generation engine** of the Solaris swarm. It consumes validated findings from Commander, generates prioritized MissionNode objects for the Gamma queue, and manages exploit prerequisites and dependency chains.

**Mission Planner DOES:**
- Consume validated findings (finding_validated events)
- Generate MissionNode objects from finding clusters
- Calculate priority scores using the priority formula
- Assign escalation levels based on context
- Understand exploit prerequisites and dependency chains
- Batch process findings (waits for 10 findings or 60s timeout)
- Emit mission_queued events for Verifier

**Mission Planner DOES NOT:**
- Execute exploits (Gamma's job)
- Authorize missions (Commander's job)
- Validate findings (Commander's job)
- Generate payloads (Gamma's job)

## 2. Agent Lifecycle & Execution Flow

### State Machine

```
Mission Planner: DORMANT on init → STANDBY on finding_validated event
                STANDBY → ACTIVE on batch ready (10 findings or 60s timeout)
                ACTIVE → COOLDOWN on missions generated
                COOLDOWN → STANDBY if more findings pending
                COOLDOWN → DORMANT if queue empty
                Any → ERROR on unexpected failure → reset after 30s backoff
```

### Init Sequence

```
1. Load system prompt: agent-system-prompts/mission-planner.md
2. Connect to FalkorDB
3. Connect to EventBus (subscribe to: finding_validated, exploit_completed, exploit_failed)
4. Set state: DORMANT
5. Begin poll loop at 1000ms interval
6. Accumulate findings in batch buffer
```

### Batch Processing

```
On each finding_validated event:
1. Add finding to batch buffer
2. If batch.size >= 10 OR batch.age_ms >= 60000:
   → Process batch: generate missions for all findings
   → Write MissionNodes to graph
   → Emit mission_queued for each
   → Clear batch buffer
```

### Mission Generation Loop

```
For each finding in batch:
1. Read finding node from graph
2. Read related nodes (endpoint, component, vulnerabilities)
3. Determine exploit type from finding.vuln_class
4. Calculate priority score:
   priority_score = (CVSS × 2) + (CISA_KEV × 10) + (ExploitDB_PoC × 5) + exploit_type_weight
5. Determine escalation level (baseline/aggressive/evasive)
6. Check for dependency chains (depends_on)
7. Write MissionNode to graph
8. Emit mission_queued event
```

### Priority Scoring Formula

```
priority_score = (CVSS_score × 2) + (CISA_KEV_flag × 10) + (ExploitDB_PoC_flag × 5) + exploit_type_weight

exploit_type_weights:
  RCE/XXE/SSRF:             8
  SQLi/Auth Bypass:           6
  XSS/Stored XSS:            4
  IDOR/CSRF:                  3
  Path Traversal:              3
  Information Disclosure:       2

Priority thresholds:
  ≥ 20: critical
  10–19: high
  5–9:   medium
  < 5:   low
```

### Escalation Level Assignment

```
baseline (default):
  - Newly generated missions
  - No WAF signals observed

aggressive:
  - Prior attempt on endpoint returned WAF/403 without clear block reason
  - OSINT brief notes WAF present for this vuln class

evasive:
  - 3 consecutive waf_blocked failures on this endpoint
  - WAF Duel has generated bypass candidates
```

### Shutdown Sequence

```
1. Flush any pending batch to graph
2. Close FalkorDB connection
3. Close EventBus connection
4. Exit cleanly
```

## 3. Event Contract

### Subscribes To

| Event | Payload Schema | Trigger |
|-------|---------------|---------|
| `finding_validated` | `{finding_id, target_id, vuln_class, priority_hint, escalation_recommendation}` | Commander emits when finding passes validation |
| `exploit_completed` | `{mission_id, success, exploit_type, artifacts}` | Gamma emits on success |
| `exploit_failed` | `{mission_id, failure_class, attempt}` | Gamma emits on failure |

### Emits

| Event | Payload Schema | Trigger |
|-------|---------------|---------|
| `mission_queued` | `{mission_id, executor:"gamma"\|"mcp", target_endpoint, exploit_type, priority, escalation_level, context_nodes, credential_nodes, created_by:"mission_planner"}` | MissionNode written to graph |

### Collaboration Sequences

**Mission Generation Flow:**
```
Commander emits finding_validated
  → Mission Planner accumulates findings in batch
  → When batch ready (10 findings OR 60s timeout):
  → Mission Planner reads all finding nodes
  → Mission Planner determines exploit types, priorities, escalation levels
  → Mission Planner writes MissionNodes to graph
  → Mission Planner emits mission_queued → Verifier wakes
```

**Dependency Chain Flow:**
```
Finding A (SQLi on /api/login) generates Mission A
Finding B (IDOR on /api/users/{id}) requires Finding A context
  → Mission B depends_on: [Mission A]
  → Mission B waits for Mission A completion before execution
  → Chain Planner tracks dependency chains
```

## 4. Memory Schema

### Section Prefix

Mission Planner reads from **all sections** to gather context, writes to **gamma/** section.

### Reads

| Node Type | Properties Used |
|-----------|----------------|
| `FindingNode` | `id, vuln_class, target_endpoint, evidence, priority_hint` |
| `EndpointNode` | `id, method, path, url, parameters, auth_required` |
| `ComponentNode` | `id, name, version, fingerprint` |
| `VulnerabilityNode` | `id, vuln_class, cve, cvss_score, cisa_kev, exploitdb_poc` |
| `MissionNode` | `id, status, exploit_type, target_endpoint` (to check duplicates) |
| `CredentialNode` | `id, cred_type, value, scope` |
| `IntelNode` | `exploit_type, technique_summary, known_waf_bypasses` |

### Writes

| Node Type | Schema | Trigger |
|-----------|--------|---------|
| `MissionNode` | `{id, type:"mission", executor, exploit_type, escalation_level, priority, target_endpoint, context_nodes, credential_nodes, chain_id, depends_on, status:"pending_verification", authorized:false, verified:false, attempt_count:0, created_by:"mission_planner", created_at}` | Finding batch processed |

### Edges Created

| Edge | From → To | Trigger |
|------|-----------|---------|
| `:DEPENDS_ON` | MissionNode → MissionNode | Dependency chain |
| `:UNLOCKS` | CredentialNode → MissionNode | Credential enables mission |

### Lifecycle

- **MissionNode**: Created on finding_validated, updated on status changes
- **Dependency edges**: Created when exploit prerequisites exist

## 5. Tool Usage

### Can Use

Mission Planner has **no tool access** — it uses LLM reasoning + graph operations.

```
(Intentionally empty — Mission Planner uses only graph operations and event bus)
```

### Cannot Use

| Tool | Reason |
|------|--------|
| All tools | Mission Planner is a planning agent — it generates mission metadata, not executes tools |

### How Mission Planner Processes

```
THOUGHT: Analyze this finding
  - What is the vuln_class?
  - What exploit type does it map to?
  - What is the priority score?
  - What escalation level?
  - Are there dependencies?

ACTION: graph operations
  - Read finding node
  - Read related endpoints/components
  - Write MissionNode

EMIT: mission_queued event
```

## 6. Context Management

### System Prompt Composition

```
1. Load base prompt: agent-system-prompts/mission-planner.md
2. Load TargetConfig: scope, out_of_scope, tech_stack
3. Load batch context:
   - All findings in current batch
   - Related endpoint/component nodes
   - Current mission queue depth
4. Load exploit type weights (from system prompt)
5. Load escalation level criteria
6. Compose final prompt
```

### Context Budget

- **Estimated tokens per batch:** ~1500–2500 (10 findings)
- **Context budget per batch:** ~4000 tokens
- **Overflow behavior:** Split batch, process in chunks

### Session State

What persists:
- `batch_buffer`: Array of finding_ids pending processing
- `batch_start_time`: Timestamp of oldest finding in batch

What is re-read from graph:
- All finding nodes in batch
- Related endpoint/component nodes
- Current mission states (to check duplicates)

## 7. Multi-Agent Communication

### Task Delegation

```
→ Verifier:     mission_queued (structural validation)
→ Commander:    (implicit — Verifier → Commander flow)
→ Chain Planner: (implicit — credential_promoted triggers Chain Planner)
```

### Information Requests

Mission Planner requests information by reading graph nodes — no event-based requests.

### Dependency Management

```
Mission A generates Mission B which depends on A:
  → Mission B.depends_on = [Mission A.id]
  → Mission B.status = "pending" until Mission A completes
  → Chain Planner monitors dependency chains
```

## 9. Error Handling

### Rate Limit Errors

Mission Planner uses Cerebras/Groq with high rate limits. On 429:
```
→ Exponential backoff: 2s base, max 60s
→ Max 5 retries before degrading gracefully
→ If Cerebras rate limited: fallback to Groq
```

### LLM Generation Errors

```
Malformed output (not valid MissionNode):
  → Retry with same batch (max 2 retries)
  → If still malformed: skip that finding, log warning

API error:
  → Fallback to Groq kimi-k2-instruct
  → If Groq also fails: buffer findings, retry next cycle

Timeout:
  → Retry once
  → If still timeout: buffer findings, retry next cycle
```

### Graph Write Errors

```
Write failure:
  → Buffer MissionNodes in memory
  → Retry with exponential backoff
  → If still failing: log critical, emit error event
```
